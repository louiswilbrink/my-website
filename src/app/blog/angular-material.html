Layout and Theming with Angular Material

Date: March 2, 2015

Last year, Google announced Material Design and began to implement this design philosophy in all of its apps. If you use Google Inbox or Maps, you've already seen what a Material Design app looks and feels like.

Angular Material brings these UI components into the Angular world. I've written two apps using only Angular-Material components and have been pleased with the results: the apps are simple and intuitive. Plus, you get to leverage the fact that your users (if they use Google Apps) will already be familiar with the interactive components on the page

To get started, install Angular 1.3.x (required) and Angular-Material using your favorite method (bower, cdn, etc). Or you can have all of that taken care of for you by installing the following generator:

npm install generator-gulp-angular
yo gulp-angular myApp

Make sure to upgrade to the latest release on GitHub, not simply the offical pre-release version (0.6.0). The docs reflect the newest commits to the repo, as do all of the examples in this blog post.

You can also use this system to make your app responsive. For example, layout-align-sm="column" will stack all child nodes on top of each other if the device screen is below 600px. layout-align-gt-sm="row" will set keep all child nodes in a row when the device screen is above 600px. Using the range of options, you can make sure your app behaves correctly across all device widths.

What is really great about this layout system is that it enforces a separation of concerns when styling a page: attributes take care of layout, while classes take care of element styles. Following this structure, you will have an easier time debugging layout or style issues by knowing where to look in the code (templates or stylesheets). It also makes sense that layout information is visible in the DOM since it describes the behavior of an element relative to it's parent, siblings, and children. Element styles that do not depend on its surrounding relatives can be kept in the stylesheet.

Programmable CSS on the backend (using SASS or Less) made CSS more sane and maintainable, so continuing this process on the frontend can only be a good thing. I think that having the ability to programmatically style on the frontend has some creative applications, and is at least more flexible.
